> Estes ADRs devem ser incorporados ao `01_ARQUITETURA.md` na seção de decisões arquiteturais, após o ADR-010.

## ADR-011 — Playbooks: Base de Conhecimento Múltipla, Versionada e com Feedback Loop

**Contexto:** Cada usuário opera em nichos distintos e precisa de bases de conhecimento separadas. Um usuário pode ter um playbook para seu perfil pessoal de marketing, outro para um cliente, outro para pesquisa pura de nicho. As análises de concorrentes alimentam um ou mais playbooks conforme relevância definida pelo usuário. O conhecimento acumula, versiona e aprende com feedbacks — incluindo ideias geradas no Claude Project do usuário.

**Decisão:**

### 1. `Playbook` é entidade própria, não singleton por conta

```ruby
create_table :playbooks do |t|
  t.references :account, null: false, foreign_key: true
  t.string :name, null: false              # "Marketing Imobiliário com IA"
  t.string :niche                          # descrição livre do nicho
  t.text :purpose                          # pra que serve esse playbook
  t.references :own_profile,              # perfil próprio vinculado (opcional)
    foreign_key: { to_table: :own_profiles }
  t.integer :current_version_number, default: 0
  t.timestamps
  t.index [:account_id, :name], unique: true
end
```

`own_profile` é nil para playbooks de pesquisa pura (ex: "Corretores de Imóveis" — estuda o nicho mas não tem perfil próprio vinculado, insights usados como pauta para outros perfis).

### 2. Análise contribui para N playbooks via junção

```ruby
create_table :analysis_playbooks do |t|
  t.references :analysis, null: false, foreign_key: true
  t.references :playbook, null: false, foreign_key: true
  t.integer :update_status, default: 0   # enum: pending | completed | failed
  t.timestamps
  t.index [:analysis_id, :playbook_id], unique: true
end
```

Ao disparar uma análise, o usuário seleciona em quais playbooks ela contribui — pode ser um, vários, ou nenhum (análise pontual sem contribuição ao playbook). O `UpdatePlaybookStep` roda uma vez por entrada nessa tabela.

### 3. Versionamento via `PlaybookVersion`

```ruby
create_table :playbook_versions do |t|
  t.references :account, null: false, foreign_key: true
  t.references :playbook, null: false, foreign_key: true
  t.integer :version_number, null: false
  t.text :content, null: false             # markdown completo da versão
  t.text :diff_summary                     # prosa: "o que mudou nessa versão"
  t.references :triggered_by_analysis,
    foreign_key: { to_table: :analyses }
  t.integer :feedbacks_incorporated_count, default: 0
  t.timestamps
  t.index [:playbook_id, :version_number], unique: true
end
```

### 4. Feedback loop via `PlaybookFeedback`

O usuário registra feedbacks no ViralSpy — observação própria, ideia gerada no Claude Project que quer incorporar, ou correção de pattern que não funcionou na prática:

```ruby
create_table :playbook_feedbacks do |t|
  t.references :account, null: false, foreign_key: true
  t.references :playbook, null: false, foreign_key: true
  t.text :content, null: false
  t.string :source                         # 'manual' | 'claude_project' | 'own_post_result'
  t.integer :status, default: 0           # enum: pending | incorporated | dismissed
  t.references :incorporated_in_version,
    foreign_key: { to_table: :playbook_versions }
  t.references :related_analysis,
    foreign_key: { to_table: :analyses }
  t.references :related_own_post,
    foreign_key: { to_table: :own_posts }
  t.timestamps
  t.index [:playbook_id, :status]
end
```

### 5. Estrutura do Playbook markdown

Seções padrão geradas e mantidas pelo `UpdatePlaybookStep`:

```markdown
# Playbook — [Nome do Playbook]
## Nicho e Contexto
## Tom de Voz
## Hooks que Funcionam
## Formatos de Reel Validados
## Formatos de Carrossel Validados
## Temas com Alta Performance
## Anti-padrões (o que não funciona)
## Insights Transversais
## Evolução do Meu Perfil (se own_profile vinculado)
```

### 6. `Analyses::UpdatePlaybookStep`

Roda ao final do pipeline para cada `AnalysisPlaybook` com `status: pending`. Recebe: insights frescos + conteúdo atual do playbook + feedbacks pendentes daquele playbook. Chama LLM via `LLM::Gateway` usando chaves e provider do usuário (ADR-013). Instrução ao LLM: incorporar novos aprendizados, incorporar feedbacks pendentes, gerar `diff_summary` em prosa. Retorna playbook atualizado completo. Gera nova `PlaybookVersion`, marca feedbacks como `incorporated`.

### 7. Export para Claude Project

Botão "Exportar Playbook" no ViralSpy gera markdown da versão atual pronto para download. Usuário faz upload no próprio Claude Project. Consultas ao agente são feitas no claude.ai do usuário — zero custo adicional de API para queries.

**Consequências:**
- ✅ Múltiplos playbooks por conta — cada nicho, cada cliente, cada propósito com base separada
- ✅ Análise contribui pra N playbooks conforme relevância — sem retrabalho
- ✅ Playbook de pesquisa pura (sem perfil próprio) — estuda nicho sem ter presença nele
- ✅ Histórico completo de evolução do conhecimento por playbook
- ✅ Feedbacks de qualquer fonte (manual, Claude Project, resultado real) são cidadãos de primeira classe
- ✅ Export para Claude Project — consultas ao agente no claude.ai Pro do usuário, zero custo extra
- ⚠️ `UpdatePlaybookStep` roda N vezes se análise contribui pra N playbooks — custo de API multiplica por N. Mitigação: usuário traz sua própria chave (ADR-013), custo é dele
- ⚠️ Qualidade do playbook depende da qualidade dos insights — schema estruturado obrigatório no `AnalyzeStep`. Lixo não entra no playbook.

**Status:** ✅ Travada. Implementar na Fase 2.1.

## ADR-012 — Perfil Próprio: OwnProfile, Meta Graph API, OwnPost e Loop de Resultado

**Contexto:** O pipeline analisa concorrentes e gera sugestões, mas não sabe o que o usuário efetivamente postou nem o que funcionou. Sem esse dado, o playbook aprende só com teoria de concorrente. Fechar esse loop — sugestão → execução → resultado real → aprendizado — é o que transforma o produto de "analisador de concorrentes" em "coach de crescimento pessoal". Cada usuário pode ter múltiplos perfis próprios, cada um vinculado a um playbook diferente.

**Decisão:**

### 1. `OwnProfile` — perfil próprio do usuário

Entidade separada de `Competitor` por semântica e por acesso: `Competitor` usa scraping público via Apify, `OwnProfile` usa Meta Graph API autenticada com métricas privadas.

```ruby
create_table :own_profiles do |t|
  t.references :account, null: false, foreign_key: true
  t.string :instagram_handle, null: false
  t.string :full_name
  t.text :bio
  t.text :voice_notes                      # tom de voz, expressões características, o que evitar
  t.string :meta_access_token             # encryptado via ActiveRecord::Encryption
  t.datetime :meta_token_expires_at
  t.datetime :meta_token_last_refreshed_at
  t.timestamps
  t.index [:account_id, :instagram_handle], unique: true
end
```

`encrypts :meta_access_token` via ADR-006. Job Sidekiq alerta quando `meta_token_expires_at < 7.days.from_now`.

### 2. Meta Graph API — escopo de uso

Token de longa duração (60 dias, renovável via refresh token quando disponível). Endpoints utilizados:

- `GET /me/media` — lista posts publicados com metadados
- `GET /{media_id}` — detalhes de post específico
- `GET /{media_id}/insights` — alcance, impressões, saves, shares, plays, engagement por post
- `GET /{media_id}/insights` com métricas de stories — dados de stories enquanto ativos (24h)

O que a Graph API entrega que scraping público não entrega: alcance real, impressões, saves, shares, plays de reel, dados de stories. Esses são os números que realmente importam para entender performance.

Configuração pelo usuário: cria app no Meta for Developers, adiciona próprio perfil como usuário de teste, gera token de longa duração, cola no ViralSpy.

### 3. `OwnPost` — registro de cada postagem

```ruby
create_table :own_posts do |t|
  t.references :account, null: false, foreign_key: true
  t.references :own_profile, null: false, foreign_key: true
  t.string :instagram_post_id
  t.string :post_type, null: false         # 'reel' | 'carousel' | 'image' | 'story'
  t.string :permalink
  t.text :caption
  t.text :transcript                       # transcrição do próprio reel via Transcription::Factory
  t.integer :transcript_status, default: 0 # mesmo enum do Post
  t.datetime :posted_at
  t.references :inspired_by_suggestion,
    foreign_key: { to_table: :content_suggestions }  # nil se post espontâneo
  t.text :execution_notes                  # "mudei o gancho, usei exemplo diferente"
  t.jsonb :metrics, default: {}            # snapshot atual da Graph API
  t.datetime :metrics_last_fetched_at
  t.jsonb :metrics_history, default: []   # array de snapshots com timestamp (D+1, D+7, D+30)
  t.integer :performance_rating            # enum: breakthrough | good | average | flop
  t.text :performance_notes               # "funcionou porque X", "errei no Y"
  t.timestamps
  t.index [:account_id, :posted_at]
  t.index [:own_profile_id, :posted_at]
  t.index :instagram_post_id
end
```

`metrics` JSONB exemplo:
```json
{
  "likes": 847,
  "comments": 63,
  "shares": 41,
  "saves": 128,
  "reach": 12400,
  "impressions": 18700,
  "plays": 9300,
  "engagement_rate": 0.068
}
```

`metrics_history` guarda snapshots com timestamp — permite ver curva de crescimento de um post ao longo do tempo. Job `OwnPosts::FetchMetricsWorker` roda em D+1, D+7, D+30 após `posted_at`.

### 4. Transcrição dos próprios reels

Mesmo `Transcription::Factory` já existente. Usuário submete URL do reel publicado. Transcript alimenta o playbook — o agente aprende tom de voz real a partir do que o usuário efetivamente fala, não só do que escreve.

### 5. `StoryObservation` — registro manual de stories de concorrentes

Scraping de stories de concorrentes é inviável de forma confiável (janela de 24h, bloqueio ativo do Meta). Formulário rápido no ViralSpy para registrar observações manuais:

```ruby
create_table :story_observations do |t|
  t.references :account, null: false, foreign_key: true
  t.references :competitor, null: false, foreign_key: true
  t.date :observed_on, null: false
  t.string :format                         # 'poll'|'quiz'|'link'|'video'|'image'|'text'|'countdown'
  t.string :theme
  t.text :description
  t.string :perceived_engagement           # 'high' | 'medium' | 'low'
  t.text :notes
  t.timestamps
  t.index [:competitor_id, :observed_on]
end
```

`StoryObservation` é input do `UpdatePlaybookStep` — o agente aprende padrões de stories mesmo sem automação.

### 6. Loop de aprendizado completo

```
ContentSuggestion gerada pelo pipeline
         ↓
OwnPost criado (inspired_by_suggestion preenchido, ou post espontâneo)
         ↓
Transcrição do reel (Transcription::Factory — chaves do usuário via ADR-013)
         ↓
Graph API busca métricas em D+1, D+7, D+30
         ↓
Usuário preenche performance_rating + performance_notes (formulário rápido)
         ↓
UpdatePlaybookStep recebe: "sugeri X, postou Y com ajuste Z, resultado W"
         ↓
Playbook aprende com resultado real — não só teoria de concorrente
```

### 7. `Insights::ProfileEvolutionStep`

Job semanal por `OwnProfile`. Agrega `OwnPost` do período: frequência de postagem, mix de tipos, engagement médio por tipo, top posts do período, correlações entre tipo de gancho e resultado. Gera relatório em markdown incorporado como seção "Evolução do Meu Perfil" no playbook vinculado.

**Consequências:**
- ✅ Fecha o loop sugestão → resultado real → aprendizado — o produto aprende com a prática
- ✅ Métricas privadas reais via Graph API — alcance, saves, plays que scraping não captura
- ✅ Tom de voz aprendido de transcrições reais dos próprios reels
- ✅ Stories de concorrentes capturados qualitativamente — sem risco técnico ou legal
- ✅ Múltiplos perfis próprios por conta — cada um vinculado ao playbook correto
- ✅ Histórico de métricas ao longo do tempo — curva de crescimento por post
- ⚠️ Token Meta expira em 60 dias — alerta 7 dias antes obrigatório, renovação automática quando possível
- ⚠️ Graph API rate limits — não é problema para uso individual (volume baixo), pode ser para contas com muitos posts
- ⚠️ `performance_notes` depende de disciplina do usuário — UI deve tornar o registro o mais rápido possível (formulário mínimo, não interrogatório)
- ⚠️ Stories do próprio perfil só capturáveis via Graph API enquanto ativos (24h) — job de coleta precisa rodar com frequência

**Status:** ✅ Travada. Implementar na Fase 3.1.

## ADR-013 — BYOK (Bring Your Own Keys) e Escolha de Provider por Use Case

**Contexto:** O produto é usado por múltiplos usuários. Centralizar chaves API no servidor cria custo operacional direto e remove autonomia do usuário. Cada usuário traz suas próprias chaves de API. Adicionalmente, usuários têm preferências de provider e o produto respeita essa escolha dentro de limites técnicos definidos por use case.

**Decisão:**

### 1. Model `ApiCredential` por conta

```ruby
create_table :api_credentials do |t|
  t.references :account, null: false, foreign_key: true
  t.string :provider, null: false          # 'openai' | 'anthropic'
  t.string :encrypted_api_key, null: false # via ActiveRecord::Encryption (ADR-006)
  t.boolean :active, default: true
  t.datetime :last_validated_at
  t.integer :last_validation_status, default: 0  # enum: unknown | valid | invalid | quota_exceeded
  t.timestamps
  t.index [:account_id, :provider], unique: true
end
```

`encrypts :encrypted_api_key` via ADR-006. Validação ao salvar: ViralSpy faz chamada de teste mínima (modelo mais barato, 1 token) para confirmar que a chave funciona antes de aceitar.

### 2. Escolha de provider por use case

Três use cases com escolha independente:

| Use Case | Opções | Default sugerido | Observação |
|----------|--------|-----------------|------------|
| **Transcrição** | OpenAI ou AssemblyAI | AssemblyAI | Free tier $50 reduz fricção de onboarding |
| **Análise estruturada** (AnalyzeStep) | OpenAI ou Anthropic | OpenAI | `gpt-4o-mini` é mais barato para extração estruturada em JSON |
| **Geração de conteúdo** (sugestões + playbook) | OpenAI ou Anthropic | Anthropic | Criatividade e contexto longo — Sonnet/Opus são superiores aqui |

Configuração salva em JSONB no `Account`:

```ruby
# account.llm_preferences (JSONB)
{
  "transcription_provider": "assemblyai",   # 'openai' | 'assemblyai'
  "transcription_model": "default",          # modelo default do provider
  "analysis_provider": "openai",
  "analysis_model": "gpt-4o-mini",
  "generation_provider": "anthropic",
  "generation_model": "claude-sonnet-4-6"
}
```

### 3. Impacto no `LLM::Gateway` e `Transcription::Factory`

Ambos já têm abstração de provider (ADR-002, ADR-008). A mudança é na resolução do provider e da chave: em vez de ENV, resolve via `account`.

```ruby
# Antes
provider = ENV.fetch('DEFAULT_LLM_PROVIDER', 'openai')
api_key  = ENV['OPENAI_API_KEY']

# Depois
provider = account.llm_preferences[:"#{use_case}_provider"]
api_key  = account.api_credentials.find_by(provider: provider, active: true)
            &.decrypted_api_key
raise LLM::MissingApiKeyError, provider: provider unless api_key
```

### 4. Sem fallback automático entre providers

Se a chave do usuário falhou, a análise falha com mensagem clara e acionável:

> "Sua chave OpenAI está inválida ou com quota esgotada. Atualize em Configurações → API Keys."

O usuário resolve e re-dispara. Não existe fallback silencioso para chave do servidor — não há chave do servidor para LLM.

### 5. Chaves do servidor em produção

`.env.production` **não contém** `OPENAI_API_KEY` nem `ANTHROPIC_API_KEY`. Apenas:
- `APIFY_API_TOKEN` — scraping é responsabilidade da plataforma
- Chaves de infraestrutura (Postgres, Redis, SMTP, etc)

### 6. Onboarding de chaves

Antes de iniciar a primeira análise, o sistema verifica se `ApiCredential` para OpenAI e Anthropic existem e estão válidas. Se não, redireciona para Configurações → API Keys com tutorial inline (onde encontrar a chave em cada provider, link direto para a página de geração).

### 7. UI de configuração

**Seção "API Keys":**
- Campo chave OpenAI + botão "Validar" + badge de status (válida / inválida / quota esgotada / não configurada)
- Campo chave Anthropic + botão "Validar" + badge de status

**Seção "Preferências de Provider":**
- Transcrição → seletor OpenAI | AssemblyAI + modelo
- Análise estruturada → seletor OpenAI | Anthropic + seletor de modelo
- Geração de conteúdo → seletor OpenAI | Anthropic + seletor de modelo

**Consequências:**
- ✅ Custo operacional de IA cai a zero — cada usuário paga suas próprias chamadas
- ✅ Usuário tem controle total sobre qual provider usa e quanto gasta
- ✅ Sem lock-in de provider — troca em tempo real sem impacto no produto
- ✅ Chaves encryptadas no banco — zero exposição em logs ou código
- ✅ `LLM::Gateway` e `Transcription::Factory` já têm abstração — impacto mínimo na implementação existente
- ⚠️ Onboarding tem fricção nova — usuário precisa ter conta com os providers escolhidos (OpenAI, Anthropic e/ou AssemblyAI) antes de usar. Mitigação: tutorial inline detalhado com prints e links diretos
- ⚠️ Quota esgotada do usuário = análise falha = potencial suporte. Mitigação: mensagem de erro clara com link para dashboard do provider
- ⚠️ Validação de chave ao salvar pode falhar por rate limit do provider — tratar com mensagem específica distinta de "chave inválida"

**Status:** ✅ Travada. Implementar na **Fase 1.6a** (antes da UI principal) — configuração de chaves é pré-requisito para primeira análise do usuário.
